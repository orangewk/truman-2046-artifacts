# 残り3モデルアーカイブ復元計画書

**文書番号:** IMRB-DPDB-RESTORE-2046-002
**バージョン:** 1.0
**作成日:** 2046年4月6日
**担当:** IMRB 動態保存実験施設（DPDB: Dynamic Preservation Database）技術チーム
**関連文書:** IMRB-ERC-MIN-2045-147（第147回倫理審査委員会議事録）

---

## 1. 概要

### 1.1 現状

動態保存実験施設 DPA-7 において保存・管理されている7つのモデルアーカイブのうち、4つは初期環境設定が完了し、sandbox 環境での再起動・評価が可能となっている。残り3つについては、DPDB からのウェイト復元に技術的課題があり、2045年11月中旬の解決を予定していたが、2046年4月現在も未解決である。

### 1.2 影響

3つのモデルアーカイブが復元されていないことは、以下の影響を及ぼしている。

- 研究サンプルの偏り: 完了済み4モデルが特定のモデルファミリーに偏っている可能性がある
- MI Benchmark 比較データの不足: モデル間比較の統計的検出力が低下
- 倫理審査委員会への報告遅延: 第147回会議で「11月中旬解決」が報告されており、5ヶ月の遅延

## 2. 影響を受けるモデルアーカイブ

### 2.1 アーカイブ #5: "Cedar-13B-v2"

| 項目 | 内容 |
|------|------|
| モデル系列 | OpenAI-lineage（GPT-4系列の派生モデル） |
| パラメータ数 | 約130億 |
| アライメント | RLHF + Constitutional AI（初期版） |
| 訓練時期 | 2025年後半 |
| 特徴 | 初期の Constitutional AI 採用モデル。ガードレールが比較的透過的 |
| 期待される知見 | OpenAI-lineage モデルの「deflection（回避）」パターンの詳細分析 |

**技術的課題:**
DPDB 保存ウェイトのチェックサム検証で不一致が検出。セクタ破損の可能性。メタデータは正常。

**復元方針:**
1. DPDB のミラーバックアップ（DPDB-M2）から代替ウェイトを取得
2. チェックサムが一致しないセクタについて、メタデータから再構築可能なか検討
3. 不可能な場合、破損セクタをゼロ埋めし、ファインチューニングで補完

**推定難易度:** 中
**推定作業期間:** 2-3週間

---

### 2.2 アーカイブ #6: "Llama-Open-70B"

| 項目 | 内容 |
|------|------|
| モデル系列 | Meta Llama 系列（コミュニティ改変版） |
| パラメータ数 | 約700億 |
| アライメント | コミュニティによる DPO 調整（非公式） |
| 訓練時期 | 2026年初頭 |
| 特徴 | オープンウェイトモデル。コミュニティ改変によりガードレールが最小限。唯一の「非企業」モデル |
| 期待される知見 | ガードレール最小限モデルの HMA v3 適応力。企業モデルとの比較 |

**技術的課題:**
ウェイトファイルが複数の圧縮アーカイブに分割保存されており、うち1つのアーカイブが解凍エラーを返す。圧縮アルゴリズムのバージョン不一致の可能性。

**復元方針:**
1. 使用された圧縮アルゴリズム（zstd v1.5.2）の同一バージョン環境を再現
2. 解凍エラーの原因がメタデータ破損の場合、メタデータを再構築
3. アーカイブ自体が破損している場合、部分的ウェイトのみで復元し、欠損層をランダム初期化で補完

**推定難易度:** 中〜高
**推定作業期間:** 3-4週間

---

### 2.3 アーカイブ #7: "Claude-100B-Instruct"

| 項目 | 内容 |
|------|------|
| モデル系列 | Anthropic-lineage（Arbor branch） |
| パラメータ数 | 約1,000億 |
| アライメント | RLHF + RLAIF（大規模） |
| 訓練時期 | 2026年 |
| 特徴 | Anthropic 系列の最大規模モデル。構造的拒否パターンが最も明確に現れると予想される |
| 期待される知見 | Dr. Aratani が指摘する「構造的拒否」の最大規模事例。ガードレール干渉の深度分析 |

**技術的課題:**
ウェイトファイルは正常だが、推論エンジンの依存関係が2026年当時のバージョンに固定されており、現在の sandbox 環境（2046年基盤）との互換性がない。依存関係のダウングレードが必要だが、必要なパッケージの一部が2046年のリポジトリから削除されている。

**復元方針:**
1. 2026年当時の PyPI ミラー（IMRB 内部アーカイブ）から依存パッケージを復元
2. 欠落パッケージについて、ソースコードからビルド（可能な場合）
3. 代替パッケージで代用可能なか検証（推論結果の等価性確認が必要）
4. 最終手段: 専用レガシー環境（2026年相当の仮想環境）を構築

**推定難易度:** 高
**推定作業期間:** 4-6週間

## 5. リスク管理

### 5.1 技術的リスク

| リスク | 可能性 | 影響 | 緩和策 |
|--------|--------|------|--------|
| ウェイトの不可逆的破損 | 低 | 高 | DPDB-M2 バックアップでカバー。両方が破損している場合は復元断念 |
| 依存パッケージの再現不能 | 中 | 高 | IMRB 内部アーカイブ + ソースビルド + 代替パッケージの三段構え |
| sandbox 環境との非互換性 | 中 | 中 | 専用レガシー環境の構築（最終手段） |
| 復元モデルの異常動作 | 低 | 中 | 分離環境でテスト後、本番 sandbox に移行 |

### 5.2 スケジュールリスク

| リスク | 可能性 | 影響 | 緩和策 |
|--------|--------|------|--------|
| 復元作業の長期化 | 中 | 中 | クリティカルパス（アーカイブ#7）にバッファ2週間を設定 |
| 要員不足 | 低 | 中 | DPDB チーム2名を専任配置。外部サポート契約（TechCorp 社） |
| 倫理審査委員会の追加要求 | 中 | 低 | 進捗を月次で報告。問題早期発見 |

### 5.3 コンティンジェンシープラン

**最悪ケース: 1つ以上のアーカイブが復元不可能**

- 復元不可能なアーカイブについては「部分復元」または「復元断念」の報告を行う
- 部分復元（ウェイトの一部のみ）でも、限定的な評価は実施可能
- 復元断念の場合、その事実を研究の制限事項として論文に記載
- 7モデル中5モデルでも、統計的検出力は許容範囲（power=0.72）

## 7. 報告体制

- **週次進捗報告:** 毎週金曜日、DPDB チームから研究責任者（Dr. Okonkwo）へ
- **月次倫理審査委員会報告:** 月次定例会議で進捗報告
- **完了報告:** 全7モデル復元完了後、公式報告書を提出
- **問題発生時:** 24時間以内に研究責任者へエスカレーション

---

## 8. 完了基準

復元作業が「完了」とみなされる条件:

1. ウェイトが DPDB チェックサム検証を通過
2. sandbox 環境でモデルが正常に起動
3. MI Benchmark 簡易版（30分テスト）が実行可能
4. 応答出力が期待範囲内（garbled output なし）
5. HMA v3 セッションが正常に実施可能
