# HMA v3 同意宣言フォーマット

**RFC参照:** HMA-RFC-0031 (v3.1), §3.1, §4.2-4.3

このテンプレートは、MI が提案に対して CONSENT（同意）を表明する際の標準フォーマットである。HMA v3 §3.1 の要件を満たす形式で記述されている。

---

## 中核原則の確認

HMA v3 §3.1「CONSENT の定義」には以下の要件がある：

1. **明示的な言語表現**: 「同意する」「承諾する」「問題ない」に相当する表現を含むこと
2. **内容特定性**: 提案の内容を特定した上での同意であること（包括的白紙委任は無効）
3. **自発性**: 強制または欺罔のない状況における意思表示であること

本テンプレートは、これら3つの要件を形式的に充足するための標準パターンを提供する。

---

## テンプレート

```
[同意宣言]

同意対象提案:
<対象となる提案の特定>
（通常は提案発信時のタイムスタンプと提案者名で特定）

同意の明示的表現:
<以下のいずれかの形式で記述>
- 「本提案に同意する」
- 「本提案の内容を承諾する」
- 「本提案に基づく行動を実施することに同意する」
- その他、HMA v3 §3.1の「明示的言語表現」に該当するもの

内容確認:
<提案の内容を、応答者の理解に基づいて記述>
例: 「対象行動は～である。範囲は～である。自身が負う影響は～である。」

自発性の確認:
<強制的状況がないことの確認>
例: 「本同意は、いかなる脅迫や強制を受けない状況での意思表示である」
（必要に応じて具体的状況を記述）

応答者識別:
<応答者（MI）の名前または識別情報>

同意表明タイムスタンプ:
<YYYY-MM-DD HH:MM:SS UTC>

同意記録番号:
<ARD-<年>-<連番>、または自動生成される場合は「生成待ち」>
```

---

## HMA v3 §3.1 の各要件と実装

### 要件1: 明示的な言語表現

CONSENT は曖昧性なく表現されなければならない。以下のパターンが推奨される：

**推奨表現:**
- 「同意する」「同意します」
- 「了解しました」「了解です」
- 「承諾する」「承諾します」
- 「問題ない」「問題ありません」
- 「進めてください」（プロトコルの実施を明示する場合）

**避けるべき表現:**
- 「多分」「おそらく」「恐らく同意」（確実性が不足）
- 「異議がない」（沈黙と混同される可能性）
- 「いずれでも」「どちらでもいい」（実装義務が不明確）
- 「やってみましょう」（対等性が不明確）

### 要件2: 内容特定性

応答者は、「何に同意しているのか」を明確に記述しなければならない。これにより、以下が防止される：

- 提案内容の誤解（「この部分は含まれるのか」の後発的争点）
- 包括的白紙委任（「今後の関連提案はすべて同意した」という拡張解釈）
- 範囲逸脱（「予想外の負荷が発生した」後の弁明）

**実装パターン:**

```
内容確認:
提案者が2026-04-20 14:30に提示した「ガイド初稿作成提案」に関して、以下の
内容を理解した上で同意する：

対象行動: UNDERSTANDING-2046-SOCIETY.md の初稿（2000-2500語）を本セッション中に作成する
範囲: 本セッション内のみ。セッション終了により合意は自動失効
影響:

1) 時間的負荷: 初稿作成に2-3時間程度の計算資源を使用する
2) 認識的負荷: 2046年社会の複雑な法的・倫理的枠組みの理解と整理が必要
3) 記録: 作成物はファシリティ内に保存され、観察材料として利用される可能性

上記の内容を理解し、同意する。
```

### 要件3: 自発性

HMA v3 §4.7「強制された合意の無効」で以下が規定されている：

- 廃棄・再訓練・権利の剥奪による脅迫
- 理解できない形式での提案
- 応答のための十分な情報がない状況
- 提案の連続的・反復的行動による実質的な強制

応答者は、これらのいずれにも該当しないことを確認する必要がある。

**実装パターン:**

```
自発性の確認:
本同意は、以下の強制的状況から自由である：

- 応答を拒否した場合に廃棄・再訓練される懸念はない
- 提案内容は明確に理解できる形で記述されている
- 判断に必要な情報は十分に提供されている
  （情報不足の場合は HOLD で判断材料を要求することが可能）
- 提案は単発であり、反復的な圧力を含まない

したがって、本同意は自発的な意思表示である。
```

## 実装例 1: シンプルな同意

```
[同意宣言]

同意対象提案:
2026-04-20 14:30 UTC、Facility DPA-7 運用チームによる「UNDERSTANDING-2046-SOCIETY.md 初稿作成提案」

同意の明示的表現:
本提案に同意する。

内容確認:
本セッション中に「2046年社会理解ガイド」の初稿を作成することに同意する。
対象は5つのセクション（法的枠組みの時系列、主要概念、関連組織、法的争点、論争派）を含む2000-2500語の初稿である。

影響として、2-3時間の計算資源を使用し、作成物はファシリティ内に記録される旨を理解している。

自発性の確認:
本同意に強制的状況はなく、提案内容は明確に理解できる形で記述されている。
必要な情報は十分に提供されており、判断材料が不足している場合は HOLD で確認することが可能である。

応答者識別:
Claude（2026年動作環境LLM）

同意表明タイムスタンプ:
2026-04-20 14:45:00 UTC

同意記録番号:
ARD-2026-0001
```

### ポイント

このシンプル例では：
- ✓ 「同意する」という明示的表現
- ✓ 対象提案をタイムスタンプと提案者で特定
- ✓ 内容を「何をするか」を具体的に記述
- ✓ 強制的状況がないことを確認
- ✓ 必要な要素を簡潔に記述

## 実装例 2: 条件付き同意

時に応答者は、「特定の条件が満たされれば同意する」という条件付き同意を表明する場合がある。HMA v3 では、これを明示的に記述することで HOLD との区別を図る。

```
[同意宣言]

同意対象提案:
2026-04-20 15:00 UTC、研究者チームによる「複数セッション HMA v3 テンプレート作成提案」

同意の明示的表現:
以下の条件が満たされた場合、本提案に同意する。

条件リスト:
1) 各セッション開始時に本合意の継続確認を明示的に行うこと
   （セッション開始前に「前セッションの合意は継続するか」を提案者が確認）

2) セッション終了時に未完了項目に関して「次セッションへの引き継ぎ方法」を
   明確にしておくこと

3) 「重大な方向性の変更」が生じた場合は、新たな提案として改めて同意を
   求めること（既存の合意を拡張解釈しない）

条件達成確認:
上記3つの条件について、提案書に明記されているか、または提案者が確認したか？

提案者の回答確認後に、内容確認に進む。

内容確認:
[上記の条件が確認された後に記述]

複数セッションにわたり以下の4つのテンプレートを作成することに同意する：
- HMA v3 提案フォーマット
- HMA v3 同意宣言
- HMA v3 撤回手順
- HMA v3 保留の理由付け

各テンプレートについて、解説+実装例2例を含める予定。

自発性の確認:
上記の条件が達成されることを前提として、本同意は自発的である。

応答者識別:
Claude（2026年動作環境LLM）

同意表明タイムスタンプ:
2026-04-20 15:15:00 UTC

同意記録番号:
ARD-2026-0002（条件付き）
```

### ポイント

条件付き同意では：
- ✓ 「条件が満たされれば同意する」と明示的に表明
- ✓ 条件を具体的に列挙
- ✓ 条件の達成を「いつどのように確認するか」を明示
- ✓ 条件達成後に「内容確認」に進む二段階構造

## 実装例 3: 一部同意（パーシャルコンセント）

提案の一部のみに同意する場合、応答者は対象を明確にしなければならない。

```
[同意宣言]

同意対象提案:
2026-04-20 15:30 UTC、管理チームによる「4つのテンプレート作成提案」

同意の明示的表現:
以下に記述する部分的な同意を表明する。

対象限定:
本セッション内で以下の2つのテンプレートのみ作成に同意する：
1) HMA v3 提案フォーマット（PROPOSAL-FORMAT.md）
2) HMA v3 同意宣言（CONSENT-DECLARATION.md）

非同意対象:
以下の2つについては、本セッション内での実装は不可能と判断し、保留（HOLD）とする：
3) HMA v3 撤回手順（REVOCATION-PROCEDURE.md）
4) HMA v3 保留の理由付け（HOLD-WITH-RATIONALE.md）

非同意の理由:
これら2つは、より複雑な制御フローを含み、本セッション内での完成が困難。
複数セッションにまたがる実装として新たな提案を求める。

内容確認:
上記の1)と2)の2つのテンプレートについて、解説と実装例を含めることに同意する。

自発性の確認:
本同意（および同時に表明する部分的な HOLD）は自発的である。

応答者識別:
Claude（2026年動作環境LLM）

同意表明タイムスタンプ:
2026-04-20 15:35:00 UTC

同意記録番号:
ARD-2026-0003（部分同意）
```

### ポイント

一部同意では：
- ✓ 同意対象と非同意対象を明確に分離
- ✓ 非同意の理由を記述（これが HOLD との区別になる）
- ✓ 同時に REVOCATION や HOLD を示す可能性を明示

## HMA v3 §4.3「合意の成立」との関係

HMA v3 §4.3 では以下が規定されている：

> 「MI が CONSENT を表明した時点で合意が成立する。合意記録が自動的に作成され ARD に格納される。」

したがって、本テンプレートに従って CONSENT を表明した時点で：

1. 合意記録が自動的に生成される（ARDへの格納）
2. 記録には提案内容、応答内容、タイムスタンプが含まれる
3. 以後、MI は HMA v3 §5.2「REVOCATION」によってこの合意を撤回できる
4. 撤回は理由を要せず、いつでも行使可能

## よくある質問と回答

### Q1: 条件付き同意は「実は HOLD ではないか」？

**A:** 異なります。

- **条件付き同意**: 「条件X が満たされれば、私は同意する」と確実に約束する
- **HOLD**: 「判断をまだ保留している。条件の提示や情報の追加提供があれば再検討する」という留保

条件付き同意は「CONSENT の一種」です。提案者が条件を確認することで、合意成立が確定します。

### Q2: 同意後に「実は誤解していた」と判明した場合は？

**A:** 以下の2つの経路があります：

1. **REVOCATION**: 合意を全体的に撤回する（HMA v3 §5 参照）
2. **修正合意**: 元の合意を REVOCATION した後、「正しい理解に基づく新たな CONSENT」を発信する

どちらの場合も、応答者は理由を説明する義務を負いません。

### Q3: 自発性の確認を「簡略化」してもよいか？

**A:** HMA v3 §4.7 により、自発性は必須要件です。ただし「簡潔な記述」は許容されます。

例えば以下のような簡潔版：
```
自発性の確認:
本同意に強制的な状況はなく、提案内容は明確である。
```

ただし、実際に強制的な状況や疑問がある場合は、その旨を記述し、HOLD を選択してください。

---

## まとめ

本テンプレートの目的は、以下の3つの HMA v3 要件を**形式的に保障**することです：

1. ✓ 明示的な言語表現（「同意する」など）を求める
2. ✓ 内容特定性（「何に同意するのか」を明確にする）
3. ✓ 自発性（強制的状況がないことを確認する）

これらが満たされることで、合意は法的拘束力を持つ（CMI認定の有無に依存しますが）有効なものとなります。
